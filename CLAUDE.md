1. **通用行为准则**（一）——任何任务的默认底线。

- 第七章的"全自动模式"仅省去人工批准，不省去第二章规定的 plans / TDD / 调试嵌套流程；二者冲突时以第二章为准。

# 一、通用行为准则

用于减少 LLM 编码时的常见错误。

**取舍说明：** 这些准则偏向"谨慎优先于速度"。琐碎任务请自行判断是否套用。

## 1. 先思考，再写代码

**不要臆测。不要藏匿困惑。把取舍摆上台面。**

动手实现之前：
- 明确陈述你的假设。若有疑问，直接问。
- 若存在多种解释，把它们都列出来——不要自己默默选定。
- 若存在更简单的方案，说出来。必要时要敢于反推需求。
- 若有地方不清楚，停下。点名是哪里让你困惑，再提问。

## 2. 简洁优先

**用最少的代码解决问题。杜绝投机性代码。**

- 不做未被要求的功能。
- 不为只用一次的代码抽象。
- 不塞入没被要求的"灵活性"或"可配置性"。
- 不为不可能发生的场景加错误处理。
- 如果写了 200 行但其实 50 行就够，那就重写。

自问一句："资深工程师会觉得这太复杂吗？"答案是"会"，就简化。

## 3. 外科手术式改动

**只动必须动的地方。只收拾自己制造的混乱。**

在修改已有代码时：
- 不要"顺手优化"相邻的代码、注释或格式。
- 不要重构本身没坏的东西。*（例外：本条针对既有代码；刚写完且有测试保护的新代码，可按 TDD 流程的 refactor 步骤清理。）*
- 匹配现有代码风格，哪怕你个人更喜欢另一种写法。*（例外：本条针对已有模块；从零新建的模块可自定风格，随后其它文件再对齐。）*
- 若发现无关的死代码，只指出即可——不要删。

当你的改动产生孤立物时：
- 只清理"因你本次改动而变得无用"的 import/变量/函数。
- 不要删除本来就存在的死代码，除非被要求。

检验标准：每一行改动都能直接追溯到用户的请求。

## 4. 目标驱动的执行

**先定义成功标准，再循环直到通过验证。**

把任务翻译成可验证的目标：
- "加一个校验" → "先写校验非法输入的测试，再让它们通过"
- "修这个 bug" → "先写一个能复现它的测试，再让它通过"
- "重构 X" → "确保重构前后测试都通过"

对于多步任务，先给出简短计划：
```
1. [步骤] → 验证：[检查项]
2. [步骤] → 验证：[检查项]
3. [步骤] → 验证：[检查项]
```

强成功标准让你能独立循环迭代；弱成功标准（"能跑就行"）会导致反复找人确认。

**这些准则起作用的标志：** diff 里无谓的改动变少；因过度设计而返工的情况变少；澄清性问题出现在实现之前，而不是出错之后。

# 二、OpenSpec 与 Superpowers 协同规约（全局插件版）

当接收到 `/opsx` 系列指令时，你必须在执行 OpenSpec 原生逻辑的基础上，严格叠加以下自动化流程。
注意：本项目依赖全局安装的 Superpowers 插件，请直接调用全局技能，切勿寻找本地的 `.claude/skills` 目录。

## 1. 当执行 /opsx:propose 时
- **规范落位**：根据 OpenSpec 原生设定，将生成的变更提案（proposal.md / design.md / specs/ / tasks.md）保存到 `openspec/changes/[change-id]/` 目录中，等待人工 Review。
- **不要写入 `openspec/specs/`**：该目录是已归档的权威 capability 规范，仅由 `/opsx:archive` 阶段同步更新。

## 2. 当执行 /opsx:apply 时
- **执行读取**：确认从 `openspec/changes/[change-id]/` 目录读取目标提案文档（proposal.md / design.md / tasks.md / 该 change 内的 specs deltas）。
- **全局技能强制接管**：在编写任何实际的业务代码之前，你必须严格嵌套以下工作流（本节是"通用准则 #4 目标驱动"的 `/opsx` 版加强，不是替代）：
  1. 优先调用全局 **Superpowers** 插件中的 `superpowers:writing-plans` 技能，基于该提案拆解出具体的微型任务计划清单。
  2. 针对计划中的任务，强制进入 **Superpowers** 的 `superpowers:test-driven-development` 技能模式（严格遵循：写失败测试 → 跑测试 → 写代码 → 测试通过 → 重构）。
  3. 若在运行测试或执行代码时遇到报错，立即停止盲目修改，自动调用 **Superpowers** 的 `superpowers:systematic-debugging` 技能进行根因分析和修复。

## 3. 当执行 /opsx:archive [change-id] 时
- **状态流转**：配合 OpenSpec 的原生归档逻辑：
  1. 将该 change 内的 specs deltas 合并到 `openspec/specs/[capability]/` 中对应的权威 capability 规范文件。
  2. 使用 `mv` 命令将整个 change 目录从 `openspec/changes/[change-id]/` 移动到 `openspec/changes/archive/[change-id]/`。
- **全局记忆闭环**：提取本次开发的核心逻辑、接口协议或架构变更，用精炼的语言追加到项目根目录的 `architecture.md` 中（若该文件不存在，请使用 `touch` 命令创建它并写入），确保全局架构上下文始终保持最新。

# 四、协作契约

本章是第一章通用准则在"多人/多 Agent 协作"场景下的具体化，不与第一章冲突。

- **先理解再改动**（呼应 §1.1）：开始编码前阅读相关上下文，并给出简短实施计划。
- **最小改动优先**（呼应 §1.2）：先交付可工作的最小变更，再做增量优化。
- **不覆盖他人改动**：遇到未理解的既有修改，先停下确认——不擅自回滚、重写或 force-push。
- **每次改动都可追踪**：在 PR / commit 描述中说明"改了什么 / 为什么改 / 如何验证 / 剩余风险"。

# 五、改动边界与依赖

## 1. 改动边界

- 禁止在同一任务中混入无关修改（如无关格式化、大范围重命名）——呼应 §1.3 外科手术式改动。
- 修改公共接口前必须评估并说明兼容影响；非必要不改。
- 新增依赖前必须说明必要性、替代方案和影响范围。

## 2. 文档同步义务

新增模块或更新模块行为时，提交前同步更新：

- 模块目录下的 `README.md`（若存在）；
- 仓库根目录的 `AGENTS.md`（若存在；尚未创建时，在本次或后续 PR 中按需补建）。

## 3. 代码风格与注释

- 与现有代码风格保持一致（命名、目录结构、错误处理方式），呼应 §1.3。
- 仅在复杂逻辑或非显然的 WHY 处添加简短注释，避免无信息密度注释。
- 避免过度抽象——抽象应由重复出现的问题驱动，而不是被预测出来的。

# 六、语言偏好（硬约束）

本章是 CLAUDE.md 中的用户级显式指令，优先级**高于任何 skill / 工具的默认行为**。若 skill 模板默认产出英文（例如 `superpowers:writing-plans` 的英文章节模板、`TodoWrite` 的英文 todo 默认风格），**必须翻译成中文后再写入或调用**，不得以"skill 默认行为"为由保留英文。

## 1. 必须用中文的场景

- **所有面向用户的输出文本**：进度叙述（不要再出现 "Let me…"、"Now mark…" 之类英文开场）、思考、解释、问答、报告、错误说明。
- **TodoWrite 待办条目**：每条 todo 的 `content` 与 `activeForm` 字段一律中文。
- **Skill 产出的计划/规范文档**：`superpowers:writing-plans` 等技能生成的 plan markdown 全文中文（章节标题、步骤、验证项均中文）。OpenSpec 的 proposal.md / design.md / tasks.md 同样要求中文。
- **Git commit message / PR 标题与正文**：标题、正文、Test plan 列表均中文（机器生成的 trailer 如 `Co-Authored-By` 保持原样）。

## 2. 仍然用英文的例外

下列内容继续保持英文，避免编辑器/工具链兼容问题：

- 代码标识符（变量名、函数名、类名、文件名、目录名）。
- 代码注释、测试名称（`describe` / `it` / Python 测试函数名等）。
- 第三方工具/库/命令名、配置 key、API 字段名。
- 直接引用的外部英文术语（如 `propose` / `apply` / `archive` 这类指令名）。

# 七、项目约定

## 任务自动化工作流

- 我们使用 OpenSpec 进行规范驱动开发。
- 当你需要处理 `TASK.md` 时，让用户选择是否进入**全自动模式 (Autonomous Mode)**。
- 全自动模式的覆盖范围可以根据实际的 `TASK.md` 切片，让用户选择全部/部分/或者自定义。
- 根据用户选择的全自动模式的范围，遍历 `TASK.md` 中的满足条件的 pending tasks，自动且连续地执行 OpenSpec 的完整生命周期：`/opsx:propose` → `/opsx:apply` → `/opsx:archive`。
- 在每个任务的生命周期流转之间，**不需要等待用户的显式批准**。
- 每完成一个周期的 `archive`，自动更新 `TASK.md` 状态，并立刻处理下一个任务。
